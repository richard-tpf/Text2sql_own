# 各文件详细说明

## 1. agent.py — 主入口文件

这是整个项目的入口和组装中心。除了组装 Agent 外，还负责定义 FastAPI 路由和 API 端点。

文件结构分为 5 个区块：

| 区块 | 内容 |
|------|------|
| 区块 1 | 配置 LLM — DeepSeek-R1 通过 SiliconFlow API |
| 区块 2 | 配置双 Milvus 知识库（DualKnowledgeBaseManager） |
| 区块 3 | 配置用户认证（SimpleUserResolver） |
| 区块 4 | 组装 Agent — 注册工具、设置中间件 |
| 区块 5 | 启动 FastAPI 服务 — 自定义路由、知识上传 API、模板下载 API |

关键配置项：
- `temperature=0.0`：DeepSeek-R1 是推理模型，低温度获得更稳定的输出
- `stream_responses=False`：使用非流式响应
- 替换了框架默认的 `/` 路由，使用自定义主页

定义的 API 端点：

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/` | 自定义主页 |
| GET | `/knowledge/schema` | 表结构上传页面 |
| GET | `/knowledge/business` | 业务文档上传页面 |
| GET | `/api/templates/{name}` | 下载模板文件 |
| POST | `/api/knowledge/schema/upload` | 上传单条表结构 |
| POST | `/api/knowledge/schema/upload_batch` | 批量上传表结构 |
| POST | `/api/knowledge/business/upload` | 上传单条业务文档 |
| POST | `/api/knowledge/business/upload_batch` | 批量上传业务文档 |

定义的请求/响应模型：

| 模型 | 用途 |
|------|------|
| SchemaUploadRequest | 表结构上传请求体（table_name, layer, ddl, source_tables） |
| KnowledgeUploadRequest | 业务文档上传请求体（content, knowledge_type, title） |
| KnowledgeUploadResponse | 统一的上传响应体（success, knowledge_id, message） |

## 2. knowledge_base.py — 双 Milvus 知识库核心实现

这是知识库的核心模块，包含三个层次的类：

### 数据模型

| 模型 | 说明 |
|------|------|
| SchemaItem | 表结构知识条目（table_name, layer, ddl, source_tables） |
| SchemaSearchResult | 表结构搜索结果（item + similarity_score + rank） |
| KnowledgeItem | 业务文档知识条目（content, knowledge_type, title） |
| KnowledgeSearchResult | 业务文档搜索结果（item + similarity_score + rank） |

### MilvusSchemaKnowledgeBase — 表结构专用知识库

核心方法：

| 方法 | 说明 |
|------|------|
| `_build_bilingual_text(table_name, ddl)` | 从 DDL 提取字段名和中文 COMMENT，构建中英文混合文本 |
| `_create_embedding(text)` | 将文本转为 1024 维向量 |
| `save_schema(table_name, ddl, layer, source_tables)` | 保存表结构到向量数据库 |
| `search_schema(query, limit, similarity_threshold, table_name_filter, layer_filter)` | 搜索表结构，支持按表名和层级过滤 |
| `delete_schema(knowledge_id)` | 根据 ID 删除表结构 |

Milvus 集合 schema（dw_schema_kb）：

| 字段名 | 类型 | 说明 |
|--------|------|------|
| id | VARCHAR(100) | 主键，UUID |
| embedding | FLOAT_VECTOR(1024) | 中英文混合文本向量 |
| table_name | VARCHAR(500) | 库名.表名 |
| layer | VARCHAR(20) | 所属层级：DWD/MID/DWS/ADS |
| ddl | VARCHAR(30000) | 建表语句原文 |
| source_tables | VARCHAR(5000) | 来源表列表 |
| timestamp | VARCHAR(50) | 创建时间 |

### MilvusBusinessKnowledgeBase — 业务文档专用知识库

核心方法：

| 方法 | 说明 |
|------|------|
| `save_knowledge(content, knowledge_type, title)` | 保存业务文档到向量数据库 |
| `search_knowledge(query, limit, similarity_threshold, knowledge_type)` | 搜索业务文档，支持按类型过滤 |
| `delete_knowledge(knowledge_id)` | 根据 ID 删除业务文档 |

Milvus 集合 schema（dw_business_kb）：

| 字段名 | 类型 | 说明 |
|--------|------|------|
| id | VARCHAR(100) | 主键，UUID |
| embedding | FLOAT_VECTOR(1024) | 文本向量 |
| content | VARCHAR(30000) | 知识内容 |
| knowledge_type | VARCHAR(50) | 类型：business/requirement/standard |
| title | VARCHAR(500) | 知识标题 |
| timestamp | VARCHAR(50) | 创建时间 |

### DualKnowledgeBaseManager — 双知识库管理器

统一管理两个知识库的门面类，提供：
- `save_schema()` / `search_schema()` — 代理到表结构知识库
- `save_business()` / `search_business()` — 代理到业务文档知识库
- `search_all()` — 同时搜索两个知识库，返回分类结果

支持的业务文档类型：

| knowledge_type | 中文标签 | 说明 |
|----------------|---------|------|
| business | 业务知识 | 业务规则、术语定义、指标口径 |
| requirement | 需求文档 | 需求方案、指标定义文档 |
| standard | 开发规范 | SQL 编写规范、命名规范 |

向量索引配置（两个知识库相同）：
- 索引类型：IVF_FLAT
- 度量方式：IP（内积）
- nlist：128

## 3. knowledge_tools.py — 知识库工具

提供两个工具供 Agent 在对话中使用。

### SaveKnowledgeTool

- 工具名：`save_knowledge`
- 功能：保存知识到对应知识库，通过 `save_target` 参数路由
- 参数：
  - `save_target`：目标知识库（`schema` 或 `business`）
  - 表结构字段：`table_name`, `layer`, `ddl`, `source_tables`
  - 业务文档字段：`content`, `knowledge_type`, `title`

### SearchKnowledgeTool

- 工具名：`search_knowledge`
- 功能：搜索知识库，支持跨库或定向搜索
- 参数：
  - `query`：搜索查询文本
  - `search_scope`：搜索范围（`all` / `schema` / `business`）
  - `table_name_filter`：按表名精确过滤（仅 schema）
  - `layer_filter`：按层级过滤（DWD/MID/DWS/ADS，仅 schema）
  - `knowledge_type`：按业务知识类型过滤（仅 business）
  - `limit`：最大返回数量（默认 5）

搜索结果会以 CardComponent 形式在前端展示，支持折叠/展开。

## 4. DWSystemPromptBuilder.py — 数仓开发系统提示构建器

继承自框架的 `DefaultSystemPromptBuilder`，为数仓开发场景定制系统提示。

### 系统提示组成

| 模块 | 方法 | 内容 |
|------|------|------|
| 角色定义 | `_build_role()` | 定义 Agent 为数仓开发助手，列出 5 项主要职责 |
| 分层架构 | `_build_architecture()` | DWD/MID → DWS → ADS 的依赖规则 |
| 开发流程 | `_build_dev_flow()` | 两阶段开发流程（分析需求→生成SQL） |
| Hive SQL 规范 | `_build_hive_standards()` | 语法要求、命名规范、SQL 风格 |
| 检查清单 | `_build_checklist()` | 10 项开发完成后的检查项 |
| 双知识库系统 | `_build_kb_workflow()` | 搜索和保存知识的标准工作流（动态生成） |

### 开发流程中的指标提取规范

第一阶段要求 Agent 从需求文档的指标定义表中逐行提取 7 个维度：
1. 指标名称
2. 字段性质（维度、原子指标、派生指标、复合指标）
3. 统计方式（计数、计数去重、求和、平均值、比率等）
4. 取值来源（来源表名、字段、多表关联条件）
5. 取值规则（具体取值逻辑和计算公式）
6. 展示格式（整数、百分比、日期格式等）
7. 备注（特殊处理规则）

### 检查清单 10 项

1. 表名缺少 schema 前缀
2. LEFT JOIN 空值未处理
3. 分区条件遗漏
4. 除零保护缺失
5. 字段名不一致
6. 指标口径偏差（与需求文档的取值规则和统计方式对比）
7. 关联条件错误（与需求文档的取值来源关联字段对比）
8. CTE flag 打标逻辑错误（与需求文档的取值规则对比）
9. 派生指标计算位置错误
10. 周/月表聚合方式错误

## 6. schema_upload_page.py — 表结构知识上传页面

生成表结构知识上传页面的完整 HTML，支持两种上传方式：

### Tab 1：手动输入 DDL
- 输入表名（可从 DDL 自动提取）
- 选择所属层级（DWD/MID/DWS/ADS）
- 输入建表语句
- 提交到 `/api/knowledge/schema/upload`

### Tab 2：上传 MD 文件
- 拖拽或选择 .md 文件
- 前端 JS 自动解析 MD 文件：按 `## 库名.表名` 拆分，提取 `### 所属层级`、`### 建表语句`、`### 来源表`
- 预览解析结果
- 批量上传到 `/api/knowledge/schema/upload`

## 7. business_upload_page.py — 业务文档知识上传页面

生成业务文档知识上传页面的完整 HTML，支持三种知识类型：

| 类型 | 图标 | 说明 | 占位符示例 |
|------|------|------|-----------|
| 业务知识 | 📖 | 业务规则、术语定义、指标口径 | 授信通过率 = 授信通过数 / 授信申请数 |
| 需求文档 | 📄 | 需求方案、指标定义文档 | 指标定义表格式（指标名称、字段性质、统计方式...） |
| 开发规范 | 📏 | SQL 编写规范、命名规范 | 所有表名必须带 zijie. schema 前缀 |

## 11. templates/ — 模板文件目录

### requirement_template.md — 需求文档模板

由产品同事填写，包含：
- 需求概述（名称、背景、提出人、上线日期）
- 指标定义表（7 列）：指标名称、字段性质、统计方式、取值来源、取值规则、展示格式、备注
- 其他说明

### development_template.md — 开发文档模板

由开发人员（或 AI 辅助）填写，包含：
- 需求来源
- 来源表分析（DWD/MID 层来源表 + 表关联关系）
- DWS 层设计（表结构 + DDL）
- ADS 层设计（日/周/月表结构 + DDL）
- 指标口径明细表（9 列）：指标名称、字段性质、统计方式、所属层级、取值来源、取值规则、SQL 表达式、展示格式、备注
- 待确认项

### schema_template.md — 表结构知识库导入模板

用于批量导入表结构的 MD 文件格式说明，每个表用 `## 库名.表名` 开头，下含 `### 所属层级`、`### 建表语句`、`### 来源表` 三部分。
