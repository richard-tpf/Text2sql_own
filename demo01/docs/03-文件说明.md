# 各文件详细说明

## 1. agent.py — 主入口文件

这是整个项目的入口和组装中心。它不包含任何业务逻辑，只负责：

- 创建各个组件的实例（LLM 服务、知识库、工具等）
- 将所有组件注入到 Agent 中
- 启动 FastAPI Web 服务

文件结构分为 5 个配置区块：

| 区块 | 内容 |
|------|------|
| 区块 1 | 配置 LLM — DeepSeek-R1 通过 SiliconFlow API |
| 区块 2 | 配置 Milvus 知识库 |
| 区块 3 | 配置用户认证（SimpleUserResolver） |
| 区块 4 | 组装 Agent — 注册工具、设置中间件 |
| 区块 5 | 启动 FastAPI 服务 |

关键配置项：
- `temperature=0.0`：DeepSeek-R1 是推理模型，低温度可以获得更稳定的输出
- `stream_responses=False`：使用非流式响应，兼容性更好

注册的工具及权限：

| 工具 | 功能 | 可用用户组 |
|------|------|-----------|
| SaveKnowledgeTool | 保存建表语句或业务知识到知识库 | admin, user |
| SearchKnowledgeTool | 搜索知识库中的相关知识 | admin, user |

## 2. knowledge_base.py — Milvus 知识库核心实现

这是知识库的核心模块，包含数据模型和 Milvus 操作逻辑。

### 数据模型

- `KnowledgeItem`：知识条目模型，包含 ID、内容、类型、标题、时间戳
- `KnowledgeSearchResult`：搜索结果模型，包含知识条目、相似度分数、排名

### MilvusKnowledgeBase 类

核心方法：

| 方法 | 说明 |
|------|------|
| `_get_embed_model()` | 懒加载 BAAI/bge-m3 嵌入模型 |
| `_create_embedding(text)` | 将文本转为 1024 维向量 |
| `_get_collection()` | 获取或创建 Milvus 集合 |
| `save_knowledge(content, knowledge_type, title)` | 保存知识到向量数据库 |
| `search_knowledge(query, limit, similarity_threshold, knowledge_type)` | 搜索知识库 |
| `delete_knowledge(knowledge_id)` | 根据 ID 删除知识条目 |

Milvus 集合 schema：

| 字段名 | 类型 | 说明 |
|--------|------|------|
| id | VARCHAR(100) | 主键，UUID |
| embedding | FLOAT_VECTOR(1024) | 文本向量 |
| content | VARCHAR(30000) | 知识内容 |
| knowledge_type | VARCHAR(50) | 类型：ddl / business |
| title | VARCHAR(500) | 知识标题 |
| timestamp | VARCHAR(50) | 创建时间 |

向量索引配置：
- 索引类型：IVF_FLAT（倒排文件索引，精确度高）
- 度量方式：IP（内积，适合归一化向量）
- nlist：128（聚类中心数量）

## 3. knowledge_tools.py — 知识库工具

提供两个工具供 Agent 在工作流中使用。

### SaveKnowledgeTool

- 工具名：`save_knowledge`
- 功能：保存建表语句（DDL）或业务知识到知识库
- 参数：
  - `content`：知识内容
  - `knowledge_type`：知识类型（"ddl" 或 "business"）
  - `title`：知识标题（可选）

### SearchKnowledgeTool

- 工具名：`search_knowledge`
- 功能：搜索知识库中的相关知识
- 参数：
  - `query`：搜索查询文本
  - `knowledge_type`：可选，按类型过滤
  - `limit`：最大返回数量（默认 5）

## 4. KnowledgeSystemPromptBuilder.py — 知识库系统提示构建器

继承自框架的 `DefaultSystemPromptBuilder`，为知识库场景定制中文系统提示。

### 核心逻辑

根据当前注册的工具动态生成系统提示，包含以下部分：

1. **角色定义**：告诉 LLM 它是一个智能知识库助手
2. **响应指南**：指导 LLM 如何组织回复
3. **知识库工作流指令**（根据可用工具动态生成）：
   - 如果有 `save_knowledge`：指导 LLM 如何保存知识
   - 如果有 `search_knowledge`：要求 LLM 在回答问题前先搜索知识库
   - 如果两者都有：提供完整的标准工作流和保存知识流程

## 5. NoOpAgentMemory.py — 空的 AgentMemory 实现

由于 Vanna Agent 构造函数要求 `agent_memory` 为必填参数，但本项目不需要历史记忆功能（已用独立的知识库替代），因此提供一个空实现。

所有方法都返回空结果或不执行任何操作：
- `save_tool_usage()` → 不做任何事
- `search_similar_usage()` → 返回空列表
- `save_text_memory()` → 返回空的 TextMemory
- `search_text_memories()` → 返回空列表
- `get_recent_memories()` → 返回空列表
- `delete_by_id()` → 返回 False
- `clear_memories()` → 返回 0

## 6. LoggingLifecycleHook.py — 工具调用日志钩子

继承自框架的 `LifecycleHook`，实现了两个钩子方法：

### `before_tool(tool, context)`
- 触发时机：每次工具执行之前
- 作用：打印工具对象的所有字段（工具名称、参数等）
- 输出格式：`===== 工具请求 =====`

### `after_tool(result)`
- 触发时机：每次工具执行之后
- 作用：打印工具执行结果的所有字段（返回数据、状态等）
- 输出格式：`===== 工具响应 =====`
- 返回 `None` 表示不修改结果

用途：开发调试时在终端查看每次工具调用的完整输入输出。

---

## 7. LoggingLlmMiddleware.py — LLM 请求/响应日志中间件

继承自框架的 `LlmMiddleware`，拦截 LLM 的请求和响应：

### `before_llm_request(request)`
- 触发时机：每次发送请求给 DeepSeek-R1 之前
- 作用：打印完整的 LLM 请求内容（消息列表、工具定义、温度等）
- 输出格式：`===== LLM 请求 =====`

### `after_llm_response(request, response)`
- 触发时机：收到 DeepSeek-R1 的响应之后
- 作用：打印完整的 LLM 响应内容（文本内容、工具调用、token 用量等）
- 输出格式：`===== LLM 响应 =====`

用途：调试 LLM 交互，查看发送了什么、收到了什么。
