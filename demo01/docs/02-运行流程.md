# 运行流程详解

## 一、启动流程

当执行 `python -m demo01.agent` 时，程序按以下顺序初始化：

```
1. 创建 OpenAILlmService 实例
   → 连接 SiliconFlow API（https://api.siliconflow.cn/v1）
   → 加载 DeepSeek-R1 模型

2. 创建 MilvusKnowledgeBase 实例
   → 配置 Milvus 连接参数（172.16.11.57:19530）
   → 集合名称：vanna_knowledge_base
   → 实际连接在首次使用时懒加载

3. 创建辅助组件
   → SimpleUserResolver（用户认证）
   → KnowledgeSystemPromptBuilder（知识库系统提示构建）
   → NoOpAgentMemory（空的记忆实现）
   → LoggingLifecycleHook（生命周期日志）
   → LoggingLlmMiddleware（LLM 日志中间件）

4. 注册工具到 ToolRegistry
   → SaveKnowledgeTool（保存知识）
   → SearchKnowledgeTool（搜索知识）

5. 组装 Agent 实例
   → 将以上所有组件注入 Agent

6. 启动 FastAPI 服务
   → VannaFastAPIServer 创建 FastAPI 应用
   → Uvicorn 监听 0.0.0.0:8000
```

## 二、用户请求处理流程

当用户在浏览器中发送一条消息时，完整的处理链路如下：

```
用户发送消息："订单表有哪些字段？"
         │
         ▼
┌─────────────────────────────────┐
│ 1. FastAPI 接收 HTTP 请求         │
│    路由到 ChatHandler             │
└──────────────┬──────────────────┘
               ▼
┌─────────────────────────────────┐
│ 2. 用户认证                       │
│    SimpleUserResolver 解析        │
│    Cookie 中的用户信息             │
└──────────────┬──────────────────┘
               ▼
┌─────────────────────────────────┐
│ 3. 生命周期钩子 - before_message   │
│    LoggingLifecycleHook 记录日志   │
└──────────────┬──────────────────┘
               ▼
┌─────────────────────────────────┐
│ 4. 构建系统提示                    │
│    KnowledgeSystemPromptBuilder  │
│    生成包含知识库工作流指令的提示    │
└──────────────┬──────────────────┘
               ▼
┌─────────────────────────────────┐
│ 5. LLM 中间件 - before_request    │
│    LoggingLlmMiddleware 记录      │
│    完整的请求内容                  │
└──────────────┬──────────────────┘
               ▼
┌─────────────────────────────────┐
│ 6. 调用 DeepSeek-R1              │
│    通过 SiliconFlow API 发送      │
│    请求并获取响应                  │
└──────────────┬──────────────────┘
               ▼
┌─────────────────────────────────┐
│ 7. LLM 中间件 - after_response    │
│    LoggingLlmMiddleware 记录      │
│    完整的响应内容                  │
└──────────────┬──────────────────┘
               ▼
        LLM 返回了什么？
        ┌───────┴───────┐
        ▼               ▼
   文本回复          工具调用请求
   直接返回          进入工具执行循环
   给用户                │
                        ▼
              ┌──────────────────┐
              │ 8. 工具执行循环    │
              │ （见下方详细说明）  │
              └──────────────────┘
```

## 三、工具执行循环（核心流程）

根据系统提示中的知识库工作流，交互会经历以下工具调用：

### 场景一：用户提问（搜索知识库）

```
步骤 1：搜索知识库
┌─────────────────────────────────────────────┐
│ 调用 search_knowledge                        │
│ 参数：query="订单表有哪些字段？"                │
│                                              │
│ → MilvusKnowledgeBase._create_embedding()    │
│   使用 BAAI/bge-m3 将问题转为 1024 维向量      │
│ → 在 Milvus 中进行向量相似度搜索               │
│ → 返回相关的知识条目                           │
└──────────────────────┬──────────────────────┘
                       ▼
步骤 2：LLM 结合知识生成回答
┌─────────────────────────────────────────────┐
│ LLM 收到搜索结果                              │
│ → 结合知识库中的建表语句和业务知识              │
│ → 生成准确的回答返回给用户                     │
└─────────────────────────────────────────────┘
```

### 场景二：用户保存知识

```
步骤 1：保存知识
┌─────────────────────────────────────────────┐
│ 调用 save_knowledge                          │
│ 参数：                                       │
│   content="CREATE TABLE t_order ..."         │
│   knowledge_type="ddl"                       │
│   title="t_order - 订单表"                    │
│                                              │
│ → MilvusKnowledgeBase._create_embedding()    │
│   将知识内容转为 1024 维向量                    │
│ → 存入 Milvus 知识库集合                       │
└──────────────────────┬──────────────────────┘
                       ▼
步骤 2：LLM 确认保存成功
┌─────────────────────────────────────────────┐
│ LLM 收到保存成功的结果                         │
│ → 向用户确认知识已保存                         │
└─────────────────────────────────────────────┘
```

每次工具调用前后，`LoggingLifecycleHook` 都会在终端打印工具的输入参数和返回结果，方便调试。

## 四、知识库工作原理

知识库是本项目的核心，它让 Agent 能够基于已有知识回答问题：

```
第一步：管理员保存建表语句
  → 用户提供 CREATE TABLE 语句
  → Agent 调用 save_knowledge(content=DDL, knowledge_type="ddl")
  → 嵌入模型将 DDL 转为向量
  → 存入 Milvus 知识库

第二步：管理员保存业务知识
  → 用户提供业务规则、术语定义等
  → Agent 调用 save_knowledge(content=知识, knowledge_type="business")
  → 嵌入模型将知识转为向量
  → 存入 Milvus 知识库

第三步：用户提问
  → 用户提出问题
  → Agent 调用 search_knowledge(query=问题)
  → 嵌入模型将问题转为向量
  → 在 Milvus 中进行向量相似度搜索
  → 返回最相关的知识条目
  → LLM 结合知识生成回答
```

向量相似度搜索的过程：
1. 用 BAAI/bge-m3 嵌入模型将查询文本转为 1024 维向量
2. 在 Milvus 中用内积（IP）度量找到最相似的知识向量
3. 返回相似度超过阈值（默认 0.5）的知识条目
4. LLM 参考这些知识来生成准确的回答
